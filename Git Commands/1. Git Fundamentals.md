 # Git and GitHub: A Simple Guide[cite: 1]

## What is Git?[cite: 1]
Git is a **version control system** that helps track changes in our project files over time[cite: 1]. 

### Why is it used?[cite: 1]
* **Save different versions:** You can save different versions of your project[cite: 1].
* **Rollback:** You can go back to an older version if something breaks[cite: 1].
* **Collaboration:** Multiple people can work on the same project safely[cite: 1].
* **Branching:** You can create branches for new features without damaging the main work[cite: 1].

> **Git is like a smart history system for our code.**[cite: 1]

---

## Why Git Was Created[cite: 1]

### Before Git[cite: 1]
Developers manually copied folders, leading to names like[cite: 1]:
* `project-final`[cite: 1]
* `project-final-2`[cite: 1]
* `project-final-latest-fixed`[cite: 1]

This process becomes messy and dangerous[cite: 1].

### Git Solves This By[cite: 1]
* Recording every single change[cite: 1]
* Identifying who changed what[cite: 1]
* Allowing quick rollbacks[cite: 1]
* Supporting multiple developers simultaneously[cite: 1]

*Note: Git was created by Linus Torvalds in 2005 for managing the Linux source code.*[cite: 1]

---

## Core Concept: How Git Works[cite: 1]

Git tracks **snapshots**, not just individual files[cite: 1]. 

> **Example:** If you change only one file in a project and commit the changes, Git does not just track that single file[cite: 1]. Instead, Git saves the **entire project state** at that exact moment[cite: 1].

### What is a Commit?[cite: 1]
A commit is[cite: 1]:
* A checkpoint[cite: 1]
* A snapshot[cite: 1]
* A saved version of the project[cite: 1]

**Timeline Example:**[cite: 1]
`commit1 (Initial project)` $\rightarrow$ `commit2 (Added login page)` $\rightarrow$ `commit3 (Fixed bug)`[cite: 1]

Git allows you to move between these commits at any time[cite: 1].

## Git vs. GitHub[cite: 1]

| Feature | Git | GitHub |
| :--- | :--- | :--- |
| **Type** | A tool installed locally on your computer[cite: 1]. | A website and cloud platform[cite: 1]. |
| **Tracking** | Tracks changes locally[cite: 1]. | Stores repositories online[cite: 1]. |
| **Purpose** | Version control system[cite: 1]. | Sharing and hosting platform[cite: 1]. |
| **Architecture** | Distributed system[cite: 1]. | Centralized hosting service[cite: 1]. |
| **Operations** | Handles local branching and merging[cite: 1]. | Handles Pull Request (PR) reviews[cite: 1]. |
| **History** | Maintains local history and rollbacks[cite: 1]. | Hosts local code online so anyone can see and contribute[cite: 1]. |
| **Extras** | Works completely offline[cite: 1]. | Includes issue tracking, access control, and code reviews[cite: 1]. |

## Key Terms and Architecture[cite: 1]

### Repository (Repo)[cite: 1]
This is your project folder[cite: 1]. A repo contains all your project files and a hidden `.git` folder that tracks every single change made over time[cite: 1].

The hidden `.git` folder contains[cite: 1]:
* History[cite: 1]
* Git objects[cite: 1]
* Commits[cite: 1]
* Configuration[cite: 1]
* Branches[cite: 1]
* Metadata[cite: 1]

* **Local Repo:** Located entirely on your personal computer[cite: 1].
* **Remote Repo:** Hosted on the internet or a cloud server (like GitHub, GitLab, or Bitbucket) so others can access it[cite: 1].

### Working Directory[cite: 1]
This is the current state of the files you are actively editing on your computer[cite: 1]. It represents your sandbox environment where changes haven't been saved to Git history yet[cite: 1].

### Staging Area[cite: 1]
Think of this as a waiting room[cite: 1]. Before you save a snapshot of your code, you must explicitly pick which files (or specific changes) you want to include[cite: 1]. This process is called **staging**[cite: 1].

### Commit Details[cite: 1]
A saved snapshot of your project at a specific point in time[cite: 1]. When you commit, you write a permanent record of your changes[cite: 1].
* Every commit gets a **unique ID** (a long string of numbers and letters called a SHA)[cite: 1].
* Every commit requires a **commit message** explaining what you changed and why[cite: 1].

**Each commit contains:**[cite: 1]
* Commit message[cite: 1]
* Author details[cite: 1]
* Timestamp[cite: 1]
* References to file snapshots[cite: 1]

### HEAD[cite: 1]
A pointer that tells Git which branch and commit you are currently looking at on your screen[cite: 1].

### Stash[cite: 1]
A temporary clipboard[cite: 1]. If you are halfway through a feature but suddenly need to switch to an urgent bug fix, you can stash your unfinished work to clean your working directory without making an incomplete commit[cite: 1]. You can pop it back out later[cite: 1].

## Branching and Merging[cite: 1]

One of Git's greatest strengths is allowing multiple people to work on the same project simultaneously without stepping on each other's toes[cite: 1].

### Branch[cite: 1]
A parallel timeline for your code[cite: 1]. The default timeline is usually called `main` or `master`[cite: 1]. If you want to build a new feature, you create a branch to write code freely without messing up the stable, working version of the project[cite: 1].

### Merge[cite: 1]
The act of combining two branches together[cite: 1]. Once your new feature is finished and tested on your side-branch, you merge it back into the `main` branch to combine your work[cite: 1].

### Merge Conflict[cite: 1]
A roadblock when Git gets confused[cite: 1]. If you and your teammate edit the exact same line of the same file on different branches and try to merge them, Git won't know whose version to keep[cite: 1]. It pauses the merge and asks you to manually choose the correct code[cite: 1].

## Common Git & GitHub Actions[cite: 1]

* **Clone:** Downloading a complete copy of a remote repository from the cloud onto your local computer for the first time[cite: 1].
* **Fetch:** Asking the remote server, *"Hey, has anyone else uploaded new changes?"*[cite: 1] Git downloads the tracking data so you can see what your team has done, but it does not alter your local working files yet[cite: 1].
* **Pull:** Fetching changes and immediately blending them into your current local branch[cite: 1]. This updates your local code with your team's latest work[cite: 1].
* **Push:** Sending your local commits up to the remote repository (GitHub, GitLab, etc.) so your team can see and use your updated code[cite: 1].
* **Fork:** Creating a personal copy of someone else's remote repository on your own cloud account[cite: 1]. This allows you to freely experiment with open-source projects without affecting the original owner's code[cite: 1].
* **Pull Request (PR):** A formal proposal to merge your branch into the main project on platforms like GitHub[cite: 1]. A PR opens a discussion page where teammates can review your code, leave feedback, and run automated tests before giving the green light to merge[cite: 1].
* **Cherry-pick:** The act of hand-picking a single commit from one branch and applying it directly to another without merging the entire branch[cite: 1].
* **Gitignore:** A special text file in a repository named `.gitignore`[cite: 1]. You list files or folders inside it (like `.env`, passwords, database files) that you want Git to completely ignore so they are never accidentally tracked or pushed to the cloud[cite: 1].
